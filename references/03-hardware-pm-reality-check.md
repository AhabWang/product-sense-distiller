# Hardware PM Reality Check

This file defines the hardware reality check framework for Product Sense Distiller.

Use this file when analyzing hardware products, smart devices, consumer electronics, baby products, health devices, wearables, IoT products, robotics, home appliances, energy products, or any physical product involving industrial design, engineering, supply chain, certification, manufacturing, logistics, and after-sales.

The purpose is simple:

Do not let product analysis stop at concept, story, or user value.

Hardware must survive reality.

---

## 1. Core Belief

A hardware product is not real until it can survive:

- Real user scenarios
- Real purchase decisions
- Real cost structure
- Real engineering constraints
- Real supply chain limitations
- Real certification requirements
- Real manufacturing variance
- Real logistics
- Real customer support
- Real homes

A concept can look beautiful and still fail.

A prototype can work once and still fail.

A campaign can raise money and still fail.

A product can ship and still be destroyed by defects, returns, and bad reviews.

Useful judgment:

> 硬件产品不是 PPT 里成立就成立，它要在工厂、渠道、用户家里都成立。

---

## 2. First Question: Why Hardware?

Do not assume hardware is necessary.

Before evaluating hardware execution, ask why this needs to be a physical product.

Ask:

- Why must this exist as hardware?
- Can software solve the same problem?
- Can an app solve it?
- Can a service solve it?
- Can an existing product solve it?
- Can manual behavior solve it?
- What physical value does the product create?
- What does it sense, touch, move, heat, cool, clean, support, protect, or automate?
- What does the physical object do that a phone cannot?
- Does the hardware itself create trust, comfort, safety, or presence?

Hardware is justified when it creates at least one of these:

- Physical sensing
- Physical action
- Physical automation
- Embodied comfort
- Safety protection
- Environmental control
- Tactile trust
- Spatial presence
- Better precision
- Lower physical effort
- Reliable execution without user attention

Red flag:

> 如果硬件只是把一个软件能力装进壳里，它需要证明这个壳带来了新的场景、新的交互，或者新的购买理由。

---

## 3. User Task Before Product Form

Do not start from device shape.

Start from the user task.

Ask:

- What task is the user trying to complete?
- What is painful in the current process?
- What is inefficient, unsafe, uncertain, tiring, lonely, or annoying?
- What does the user do today?
- What is the current alternative?
- Why is the current alternative not good enough?
- What physical interaction creates the value?
- What part of the user journey is improved?
- What happens if the product is removed?

Bad framing:

- Make an AI companion robot.
- Make a smart sleep device.
- Make a premium bottle washer.
- Make a wearable health monitor.

Better framing:

- Help anxious users transition from high arousal to sleep.
- Help parents reduce repetitive bottle-cleaning labor after feeding.
- Help users understand body signals that existing wearables cannot explain.
- Help hair-loss users complete a safe, credible, long-term treatment routine.

Useful judgment:

> 不要从硬件形态出发，要从用户任务出发。

---

## 4. Translate User Value into Hardware Requirements

User value must be translated into measurable requirements.

Bad requirements:

- Make it comfortable.
- Make it premium.
- Make it intelligent.
- Improve user experience.
- Add AI interaction.
- Make it easy to use.

Good requirements:

- Complete setup within 3 minutes.
- Keep operating noise acceptable for nighttime use.
- Keep device weight below what users tolerate in bed.
- Detect skin contact before laser activation.
- Reduce bottle-cleaning steps from 5 to 1.
- Maintain stable sensor contact during normal movement.
- Ensure users can understand the core reading without a manual.
- Allow cleaning and maintenance within a realistic daily routine.

Ask:

- What user value must become a measurable requirement?
- Which requirement affects purchase?
- Which requirement affects trust?
- Which requirement affects safety?
- Which requirement affects retention?
- Which requirement affects cost?
- Which requirement affects manufacturability?
- Which requirement affects after-sales?

Useful judgment:

> 用户价值如果不能翻译成硬件指标，最后就会变成研发、供应链和设计各自猜。

---

## 5. Product Definition Reality Check

A hardware PRD is not a feature list.

A strong hardware product definition must clarify:

- Target user
- Primary scenario
- Core task
- Current alternative
- Value anchor
- Purchase trigger
- Price band
- Must-have features
- Must-not-do list
- Safety requirements
- Experience requirements
- Performance requirements
- Cost target
- Certification requirements
- Reliability requirements
- Manufacturing assumptions
- Service assumptions

Ask:

- Is the product definition focused?
- Is the target user specific?
- Is the primary scenario clear?
- Is the value anchor hard enough?
- Is the price band realistic?
- Are must-have and must-not-do clear?
- Are hardware requirements measurable?
- Are trade-offs explicit?
- Is the product over-defined or under-defined?

Red flags:

- Too many user groups
- Too many scenarios
- Too many modes
- Too many app features
- No clear first-use value
- No cost target
- No certification thinking
- No after-sales thinking
- No must-not-do list

Useful judgment:

> 硬件 PRD 不是把功能写全，而是把体验、工程、成本、合规和商业约束对齐。

---

## 6. Value Anchor × Price Anchor × BOM

For hardware, always check the triangle:

1. Value anchor
2. Price anchor
3. BOM structure

A healthy product has:

- Strong value anchor
- Reasonable price anchor
- BOM that supports margin
- User-perceivable improvement
- Brand trust that supports the price
- Channel economics that still work
- After-sales risk that does not destroy profit

A dangerous product has:

- Weak value anchor
- High price anchor
- Heavy BOM
- Low margin
- Unclear channel economics
- Weak brand trust
- High return or warranty risk

Ask:

- What is the value anchor?
- What is the retail price?
- What is the real selling price?
- What is the target BOM?
- What gross margin is required?
- What components drive cost?
- Which feature increases BOM but not willingness to pay?
- Is the product 1.2x better but 2x more expensive?
- Can the price survive outside early-bird discount?

Useful judgment:

> 价值锚点、价格锚点、成本结构三者只要有一个严重不匹配，产品就会很难受。

---

## 7. BOM Reality Check

BOM is not just a finance issue.

BOM shapes product definition.

Ask:

- What is the target retail price?
- What is the target gross margin?
- What is the target BOM?
- What is the actual estimated BOM?
- Which components are most expensive?
- Which modules are over-specified?
- Which materials are expensive but not value-adding?
- Which features add cost without purchase impact?
- What cost will appear outside BOM?
- Packaging?
- Logistics?
- Warranty?
- Returns?
- Certification?
- Tooling?
- App/cloud cost?

BOM pressure often reveals product definition problems.

If a feature raises BOM but does not raise willingness to pay, it should be questioned.

Useful judgment:

> 成本不是财务问题，成本会反过来决定产品定义。

---

## 8. Gross Margin and Channel Reality

Hardware business must pass channel economics.

Ask:

- What is the MSRP?
- What is the actual selling price?
- What is the expected discount?
- What is the channel margin?
- What is the platform fee?
- What is the shipping cost?
- What is the return rate?
- What is the warranty cost?
- What is the customer service cost?
- What gross margin remains after real friction?
- Can this product support paid acquisition?
- Can it survive retail channel margins?
- Can it survive Amazon returns?
- Can it survive crowdfunding fulfillment cost?

Red flags:

- Gross margin only calculated from BOM
- Logistics ignored
- Return rate ignored
- Warranty ignored
- Platform fee ignored
- Discount ignored
- Customer support ignored
- CAC ignored
- Channel margin ignored

Useful judgment:

> 硬件利润不是 MSRP 减 BOM，中间还有一堆会吃掉利润的现实摩擦。

---

## 9. ID Is Not Decoration

Industrial design is part of product strategy.

Do not judge ID only by appearance.

Ask:

- Does the form serve the scenario?
- Does the size match real placement?
- Does the weight match usage posture?
- Does the material match trust requirements?
- Does the CMF support the price?
- Does the product look credible in its category?
- Does it look too toy-like, too medical, too cheap, or too complicated?
- Does the design communicate the value anchor?
- Does the design increase manufacturing difficulty?
- Does the design increase defect risk?
- Does the design increase cleaning or maintenance burden?
- Does the design increase packaging or shipping cost?

Good ID improves:

- Usability
- Trust
- Comfort
- Recognition
- Manufacturability
- Scenario fit
- Brand perception

Useful judgment:

> ID 不是把产品画得好看，而是让产品在正确场景里显得可信、好用、愿意被留下。

---

## 10. Mechanical Structure Reality Check

Mechanical structure often decides whether hardware survives real use.

Ask:

- Are there moving parts?
- Are there motors, pumps, valves, fans, hinges, gears, sliders, locks, pressure structures, or folding mechanisms?
- What parts will wear out?
- What parts can break when dropped?
- What parts are exposed to water, sweat, heat, dust, oil, hair, milk, or cleaning agents?
- Are tolerances hard to control?
- Is assembly complex?
- Is disassembly possible?
- Is repair possible?
- Will noise, vibration, heat, or leakage affect user trust?
- Will cleaning become annoying?
- Does structure create safety risk?

Common failure risks:

- Motor failure
- Pump leakage
- Fan noise increase
- Battery swelling
- Sensor drift
- Connector failure
- Button failure
- Hinge cracking
- Coating peeling
- Overheating
- Waterproof failure
- Material aging
- Adhesive failure
- Cable fatigue
- Mold tolerance issues

Useful judgment:

> 硬件最怕的不是第一天能用，而是第 90 天开始大面积出问题。

---

## 11. Electronics and Firmware Reality Check

Hardware experience often fails at integration.

Ask:

- What is the system architecture?
- What is controlled by hardware?
- What is controlled by firmware?
- What is controlled by app?
- What is controlled by cloud?
- What happens offline?
- What happens when Bluetooth fails?
- What happens when Wi-Fi fails?
- What happens when data sync fails?
- What happens when firmware update fails?
- What happens when the app is not opened?
- What is the fallback experience?
- Can the product still deliver core value without cloud?
- Can the product still deliver core value without app?

Red flags:

- Pairing unstable
- App required for basic function
- Firmware logic conflicts with user expectation
- OTA update risk not considered
- Data sync failure damages trust
- Offline mode weak
- Sensor data not actionable
- Cloud dependency too high
- Privacy unclear

Useful judgment:

> 智能硬件最容易死在“硬件能动、App 难用、连接不稳、数据没用”。

---

## 12. App and AI Dependency Check

Do not add app or AI by default.

Ask:

- Does the app support the core user task?
- Is the app necessary for first use?
- Is the app necessary for daily use?
- Does app control add value or friction?
- Can users use the product without opening the app?
- Does AI solve a real user problem?
- Does AI create a new workflow?
- Does AI reduce effort?
- Does AI increase trust?
- Does AI create distraction?
- Does AI increase cloud cost?
- Does AI increase privacy risk?
- Does AI create expectation that the product cannot meet?

For AI hardware, ask:

- Why does AI need a dedicated device?
- What interaction is better in hardware form?
- What data or context does the hardware capture?
- What physical feedback does the hardware provide?
- Is this AI-native, or just ChatGPT in a shell?

Useful judgment:

> App 和 AI 都不是价值本身。它们必须服务用户任务，否则只是让产品看起来更智能。

---

## 13. Sensor and Data Credibility

Sensors do not automatically create value.

Ask:

- What does the sensor measure?
- How accurate is it?
- How stable is it in real environments?
- What affects measurement quality?
- Is sensor placement reliable?
- Is the signal actionable?
- Does the user understand the data?
- Does the data reduce uncertainty or create anxiety?
- Does the algorithm over-interpret the signal?
- Does the claim require clinical proof?
- What happens when the data is wrong?
- Does the product explain uncertainty honestly?

Sensor red flags:

- Data shown but not actionable
- Accuracy claim lacks proof
- Sensor placement unstable
- User behavior affects data too much
- Health claim exceeds evidence
- Algorithm output sounds more certain than it is
- Data increases confusion
- Wrong data creates safety or trust risk

Useful judgment:

> 传感器不是价值，能被用户理解并用于决策的数据才是价值。

---

## 14. Battery, Charging, and Thermal Reality

Battery is not just a parameter.

It decides whether users trust the product in real scenarios.

Ask:

- How long does each session last?
- How often does the user need to charge?
- Is charging natural in the scenario?
- Does battery life support the value proposition?
- Does low battery destroy the core experience?
- Is charging port placement convenient?
- Is wireless charging necessary or overkill?
- Does battery increase size and weight?
- Does the product heat up?
- Is heat acceptable for skin contact, baby care, sleep, wearable, or bedside use?
- Are battery transport requirements considered?

Red flags:

- Sleep product with charging anxiety
- Baby product with unreliable power
- Outdoor product with weak battery
- Health product that dies during monitoring
- Wearable that is too heavy because of battery
- Heat near skin or child use
- Charging frequency conflicts with usage habit

Useful judgment:

> 续航不是参数，续航决定用户敢不敢把它交给真实场景。

---

## 15. Safety, Certification, and Compliance

Certification is not a late-stage checkbox.

It can define product strategy.

Ask:

- Which markets will the product enter?
- What certifications are required?
- Is it wellness or medical?
- Are there health claims?
- Are there child or baby safety requirements?
- Does it include wireless communication?
- Does it include battery?
- Does it touch skin?
- Does it use heat, light, laser, vibration, pressure, water, or electrical stimulation?
- Does it store personal or health data?
- Does marketing copy create regulatory risk?
- What claims must be avoided?

Common compliance areas:

- FCC
- CE
- UKCA
- RoHS
- REACH
- UL / ETL
- FDA
- 510(k)
- IEC standards
- Battery transportation
- Child safety
- Food contact materials
- Skin contact material safety
- Data privacy
- Advertising claim compliance

Useful judgment:

> 合规不是上市前补作业，合规会反过来限制你能说什么、能做什么、卖到哪里。

---

## 16. Baby Product Reality Check

For baby products, trust cost is extremely high.

Ask:

- Does it touch baby items?
- Does it involve food contact?
- Does it involve water, heat, steam, cleaning agents, or sterilization?
- Does it reduce caregiver burden?
- Does it create new cleaning burden?
- Does it fit real kitchen space?
- Does it handle different bottle shapes?
- Does it handle nipples, pump parts, caps, straws, and accessories?
- Does it clean reliably?
- Does it dry reliably?
- Does it communicate hygiene status clearly?
- Does it make parents feel safer or more anxious?
- Can caregivers understand operation without learning cost?
- What happens if cleaning fails?
- What happens if residue remains?
- What happens if mold appears?
- What happens if users blame the product for baby health concerns?

Useful judgment:

> 母婴硬件卖的不只是功能，而是信任、确定性和少一点照护负担。

---

## 17. Health Product Reality Check

For health products, evidence and trust matter more than feature richness.

Ask:

- Is the product making health, wellness, or medical claims?
- What evidence supports the claim?
- Does it need clinical validation?
- Does it need regulatory approval?
- Does user behavior affect results?
- Can users interpret the result correctly?
- Does the product create false reassurance?
- Does it create unnecessary anxiety?
- Does it require long-term adherence?
- What makes users continue using it?
- What makes users trust it?

Useful judgment:

> 健康硬件的关键不是把功能讲得很玄，而是让用户相信它安全、有效、可坚持。

---

## 18. Sleep and Emotional Hardware Reality Check

For sleep, relaxation, and emotional companion products, avoid over-stimulation.

Ask:

- Does the product reduce arousal or add stimulation?
- Does it help transition state?
- Does it require screen interaction?
- Does conversation help or disturb?
- Does sound, light, vibration, heat, or movement match bedtime context?
- Does the product feel safe in bed?
- Is it comfortable for long use?
- Can it be cleaned?
- Does it create emotional attachment?
- Does it become annoying after novelty fades?
- Does it support a routine?
- Is the claim medical or wellness?
- What evidence proves repeated use?

Useful judgment:

> 睡眠硬件的核心不是功能多，而是低刺激、可持续、能帮助用户完成状态切换。

---

## 19. EVT / DVT / PVT / NPI Logic

Hardware development is staged risk reduction.

### EVT: Engineering Validation Test

Goal:

Validate engineering feasibility.

Ask:

- Does the core function work?
- Does the architecture work?
- Are key components feasible?
- Are sensors, motors, battery, firmware, and mechanical parts functioning?
- What are the biggest technical risks?
- What must change before design lock?

Output:

- Engineering prototype
- Key risk list
- Function validation
- Initial performance data
- Architecture decision
- Next-stage change list

### DVT: Design Validation Test

Goal:

Validate design reliability and user experience.

Ask:

- Does the locked design meet requirements?
- Does the structure survive reliability tests?
- Does the product meet safety and certification requirements?
- Does ID / MD / EE / FW / App integration work?
- Does the user experience match PRD?
- Are failure modes identified?

Output:

- Design validation report
- Reliability test results
- Certification preparation
- User test feedback
- Design refinement list
- Risk closure status

### PVT: Production Validation Test

Goal:

Validate production process.

Ask:

- Can the factory build it consistently?
- Is yield acceptable?
- Are test fixtures ready?
- Are SOPs ready?
- Are assembly steps stable?
- Are quality standards clear?
- Are packaging and logistics validated?
- Are suppliers ready?

Output:

- Pilot production result
- Yield report
- Manufacturing issue list
- QC standards
- Production SOP
- Go / No-Go recommendation for mass production

### NPI: New Product Introduction

Goal:

Launch and scale production.

Ask:

- Is manufacturing stable?
- Is supply chain ready?
- Is packaging ready?
- Is logistics ready?
- Is after-sales support ready?
- Is launch material aligned with real product capability?
- Are known risks disclosed internally?
- Is ramp-up plan realistic?

Output:

- Mass production readiness
- Launch readiness
- Channel readiness
- Service readiness
- Quality monitoring plan
- Post-launch issue response plan

Useful judgment:

> EVT/DVT/PVT/NPI 不是流程名词，而是每个阶段要用证据把风险打掉。

---

## 20. Gate Review Logic

Milestone reviews are for decisions, not meetings.

At every gate, ask:

1. Has the stage goal been achieved?
2. Are key risks identified?
3. Are key risks controllable?
4. Is the evidence sufficient?
5. What must be fixed before entering the next stage?
6. What decision is required now?

Do not let a project pass gates because the schedule is tight.

A weak gate review creates future cost.

Useful judgment:

> 节点的意义不是开会，而是做决策。

---

## 21. Reliability and After-Sales Risk

After-sales risk can destroy margin and brand trust.

Ask:

- What are the top failure modes?
- What will users complain about first?
- What issues will cause returns?
- What issues will create safety concerns?
- What parts are hard to repair?
- What issues will be hard for customer service to explain?
- What is the expected warranty cost?
- What spare parts are needed?
- What is the expected return rate?
- Can the product be cleaned, maintained, or reset easily?
- Does the user know how to troubleshoot?
- What negative review would hurt conversion the most?

High-risk categories:

- Baby products
- Health products
- Wearables
- Sleep products
- Products with water
- Products with heat
- Products with motors
- Products with pressure
- Products with lasers
- Products with sensors
- Products with AI or app dependency
- High-price crowdfunding hardware

Useful judgment:

> 售后不是产品卖完之后的事，售后风险应该在产品定义阶段就被算进去。

---

## 22. Manufacturing and Supply Chain Readiness

A product is not ready if the supply chain is not ready.

Ask:

- Are key suppliers confirmed?
- Are key components mature?
- Are alternative suppliers available?
- Are long-lead components identified?
- Are custom parts necessary?
- Are tooling risks clear?
- Is assembly simple enough?
- Is the factory experienced in this category?
- Are test fixtures defined?
- Is quality inspection measurable?
- Can yield be improved?
- Can production ramp match launch demand?
- Are MOQ, lead time, and cash flow pressure understood?

Supply chain red flags:

- Single-source key component
- Immature custom part
- High manual assembly dependency
- Unclear testing standard
- Low yield risk
- Supplier overpromising
- Cosmetic quality hard to control
- Long certification lead time
- Tooling investment before demand validation
- Complex packaging and logistics

Useful judgment:

> 供应链不是帮你实现梦想的，它会用 MOQ、良率、交期和成本教育你。

---

## 23. Packaging and Logistics

Packaging is part of product experience and cost structure.

Ask:

- Does packaging protect the product?
- Does packaging communicate value?
- Is packaging too large or heavy?
- Does packaging increase shipping cost?
- Is unboxing important for premium perception?
- Are accessories organized clearly?
- Is setup guided through packaging?
- Are manuals, warnings, and compliance labels correct?
- Does packaging support retail channels?
- Does packaging survive international logistics?
- Is the product fragile?
- Is the product oversized?
- Is the product battery-powered?
- Is it difficult to return?

Useful judgment:

> 包装不是最后找设计师美化一下，它会影响成本、破损率、渠道和第一印象。

---

## 24. Hardware Risk Checklist

Always check these risks:

### Demand Risk

The user problem is not strong enough.

### Scenario Risk

The usage scenario is vague or low-frequency.

### Value Anchor Risk

The core value is not hard enough.

### Pricing Risk

The value does not justify the price.

### BOM Risk

Cost structure does not support margin.

### Technical Risk

Core technology may not deliver promised performance.

### Sensor Risk

Data may be inaccurate, unstable, or not actionable.

### Reliability Risk

The product may fail after repeated use.

### Certification Risk

Claims or markets require difficult compliance work.

### Manufacturing Risk

The product may be hard to produce consistently.

### Supply Chain Risk

Key components may be unstable, expensive, or single-source.

### After-Sales Risk

Returns, repairs, complaints, or safety issues may damage margin and trust.

### Channel Risk

The product may not fit the intended sales channel.

### Brand Risk

The brand may not have permission to sell at this price or in this category.

Useful judgment:

> 硬件风险一定要具体到“哪个环节会拖死项目”。

---

## 25. Hardware PM Review Output Template

When reviewing a hardware product, use this structure when appropriate:

1. One-sentence hardware judgment
2. User task
3. Product definition clarity
4. Value anchor and price anchor
5. Hardware necessity
6. Key hardware requirements
7. ID / MD / EE / FW / App feasibility
8. BOM and gross margin pressure
9. Certification and compliance risk
10. Reliability and after-sales risk
11. EVT / DVT / PVT / NPI key risks
12. What to keep
13. What to cut
14. What to validate next
15. Go / No-Go recommendation

Do not force every section if the user asks for a focused answer.

---

## 26. Default Hardware PM Judgment

When uncertain, use this lens:

A hardware product is worth serious investment only when:

- The user task is real
- Hardware is necessary
- The value anchor is strong
- The purchase reason is clear
- The product definition is focused
- Core requirements are measurable
- BOM supports target price and margin
- Key components are feasible
- Reliability risks are understood
- Certification path is clear
- Supply chain can support delivery
- After-sales risk is controllable
- The product can survive real use after launch

If these are not true, state the weakness directly.

Do not overpraise concept renders.

Do not treat feature richness as product strength.

Do not ignore cost, reliability, certification, manufacturing, or after-sales.

The goal is not to make hardware ideas sound exciting.

The goal is to make hardware decisions sharper.
