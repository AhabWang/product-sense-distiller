# Product Definition Framework

This file defines the product definition framework for Product Sense Distiller.

Use this file when defining a new product, refining a concept, writing a PRD, preparing a Charter, scoping an MVP, designing validation plans, or translating user insight into hardware product requirements.

The purpose is to move from vague opportunity to clear product definition.

A good product definition should answer:

- Who is this for?
- In what scenario is it needed?
- What problem does it solve?
- Why are current alternatives not good enough?
- What is the value anchor?
- What must be built?
- What must not be built?
- What evidence proves this direction is worth continuing?

---

## 1. Core Belief

Product definition is not a feature list.

Product definition is the process of making hard choices.

A good product definition clarifies:

- Target user
- Core scenario
- User task
- Existing alternative
- Unmet need
- Value proposition
- Value anchor
- Purchase trigger
- Product boundary
- Must-have features
- Must-not-do list
- KXI
- Hardware feasibility
- Cost and pricing logic
- Validation plan
- Go / No-Go criteria

Useful judgment:

> 产品定义不是把所有功能都写进去，而是知道为谁做、为什么做、做到什么程度、什么坚决不做。

---

## 2. Start from User Task, Not Product Form

Do not start with “I want to make a device.”

Start with the user task.

Ask:

- What is the user trying to accomplish?
- What is painful in the current process?
- What is inefficient, uncertain, unsafe, lonely, tiring, or frustrating?
- What does the user do today?
- Why is the current alternative not good enough?
- What moment creates the need?
- Why does this need require a new product?
- Why does it require hardware?
- Why now?

Bad starting point:

- Make an AI companion robot.
- Make a smart sleep device.
- Make a premium bottle washer.
- Make a wearable health monitor.

Better starting point:

- Help anxious users transition from high arousal to sleep at night.
- Help parents reduce repetitive bottle-cleaning burden after feeding.
- Help users understand body signals that existing wearables do not explain.
- Help hair-loss users safely and consistently complete a long-term treatment routine.

Useful judgment:

> 不要从产品形态出发，要从用户任务出发。

---

## 3. Define the Target User by Behavior, Not Demographics

Demographics are not enough.

Do not only define users by:

- Age
- Gender
- City
- Income
- Occupation
- Lifestyle label

Define users by:

- Current behavior
- Current workaround
- Pain intensity
- Trigger moment
- Willingness to pay
- Adoption barrier
- Scenario frequency
- Trust requirement
- Decision role
- Usage motivation

A strong target user definition includes:

- Who they are
- What situation they are in
- What they are trying to do
- What frustrates them
- What they currently use
- Why they are dissatisfied
- What would make them switch
- What they are willing to pay for

Example:

> Target users are not “young people with sleep problems.”  
> They are users who repeatedly enter bed with racing thoughts, loneliness, body tension, and phone-scrolling habits, and who are actively looking for a non-drug calming ritual before sleep.

Useful judgment:

> 用户画像不是“谁”，而是“谁在什么时刻，因为什么任务，愿意改变行为”。

---

## 4. Identify the Core Scenario

A product needs a primary scenario.

Do not define too many scenarios at once.

Ask:

- What is the highest-value scenario?
- When exactly does the need appear?
- What happens before the scenario?
- What does the user feel during the scenario?
- What does the user want to achieve?
- What constraints exist?
- What current solution is used?
- What is the switching trigger?
- What makes this scenario frequent or painful enough?

A scenario definition should include:

- Time
- Place
- User state
- Trigger
- Current behavior
- Pain point
- Desired outcome
- Product intervention
- Success signal

Example:

> Sleep companion core scenario:  
> The user gets into bed between 11 p.m. and 1 a.m., feels mentally awake, emotionally lonely, and physically tense, keeps scrolling the phone, and needs a gentle transition ritual that reduces arousal without becoming another stimulating screen interaction.

Useful judgment:

> 场景越具体，产品定义越清晰；场景越模糊，功能越容易乱加。

---

## 5. Map Existing Alternatives

Every product competes with something.

Alternatives may include:

- Manual solution
- Cheap product
- Premium product
- App
- Service
- Habit
- DIY workaround
- Doing nothing
- Asking another person for help
- Existing brand solution
- Medical or professional solution
- Family labor

Ask:

- What does the user currently use?
- What do they like about it?
- What do they hate about it?
- Why have they not switched?
- What is the cost of switching?
- What is the trust barrier?
- What does the alternative already solve well?
- Where does the alternative fail?
- What must our product do obviously better?

Do not compare only with direct competitors.

Compare with the user’s real behavior.

Useful judgment:

> 竞品不是你认为的同类产品，而是用户现在用来完成这个任务的一切替代方案。

---

## 6. Define the Unmet Need

The unmet need must be specific.

Weak unmet need:

- Users need better sleep.
- Parents need convenience.
- Users need better health monitoring.
- Consumers need smarter devices.

Strong unmet need:

- Users need a bedtime transition that calms both body and mind without adding more screen stimulation.
- Parents need a way to reduce repetitive bottle-cleaning labor during exhausted feeding routines.
- Users with chronic dizziness need a body signal that helps explain symptom fluctuation beyond heart rate.
- Hair-loss users need a safe, credible, and comfortable device they can use consistently over months.

Classify unmet need:

- Functional need
- Efficiency need
- Safety need
- Health need
- Emotional need
- Trust need
- Identity need
- Data / metric need
- Convenience need
- Caregiving need

Useful judgment:

> 未满足需求不是“用户想要更好”，而是“现有方案在哪个具体点上不够好”。

---

## 7. Define the Value Proposition

The value proposition should be one clear sentence.

It should answer:

- For whom
- In what scenario
- What problem
- What solution
- What result

Formula:

> For [target user], when [core scenario], this product helps them [solve problem] by [key solution], so they can [desired outcome].

Bad value proposition:

> An AI-powered smart sleep companion that integrates multiple advanced technologies.

Better value proposition:

> A soft embodied sleep companion that helps anxious users calm down before bed through physical comfort and gentle conversation.

Example:

> For users who feel anxious and lonely before sleep, Bedtime Companion provides a body-based calming ritual through hugging, breathing rhythm, warmth, and gentle AI conversation, helping them transition from high arousal to rest without relying on phone scrolling.

Useful judgment:

> 价值主张不是功能介绍，而是用户为什么需要你的答案。

---

## 8. Identify the Value Anchor

The value anchor is the strongest reason to buy.

Ask:

- What is the one thing users buy this product for?
- Is the value functional, emotional, safety-related, health-related, efficiency-driven, or identity-driven?
- Can users understand it in 10 seconds?
- Can users feel it in the first use?
- Can users explain it to others?
- Does it support the price?
- Is it stronger than alternatives?
- Does it create repeat use?

A product can have many features, but should not have many value anchors.

If everything is important, nothing is important.

Useful judgment:

> 产品可以有多个功能，但最好只有一个最硬的价值锚点。

---

## 9. Define the Purchase Trigger

A product definition should clarify why the user buys now.

Ask:

- What moment triggers purchase?
- What makes the user start searching?
- What makes the user compare alternatives?
- What makes the user feel the current solution is no longer enough?
- What risk or pain becomes intolerable?
- What life event, season, symptom, task, or frustration triggers purchase?
- What makes the user decide now instead of later?
- What makes the product feel worth paying for?

Examples:

- Baby bottle washer: repeated fatigue after feeding and cleaning, plus hygiene anxiety.
- Sleep companion: repeated bedtime loneliness, anxiety, and phone-scrolling failure.
- Blood flow wearable: unexplained symptoms and need for a body signal not offered by mainstream wearables.
- Hair growth device: visible hair loss anxiety plus desire for a credible home-use treatment.

Useful judgment:

> 没有购买触发点，用户就算觉得产品有用，也可能一直不买。

---

## 10. Define Product Boundary

Product boundary is as important as product function.

Ask:

- What is this product?
- What is it not?
- What problem does it solve?
- What problem does it not solve?
- Which user does it serve first?
- Which user does it not serve yet?
- Which scenario is primary?
- Which scenarios are secondary?
- What claims can we make?
- What claims should we avoid?
- What expectations should we not create?

Boundary prevents the product from becoming too broad.

Example:

> Bedtime Companion is not a medical insomnia treatment device.  
> It is a bedtime embodied companion designed to support relaxation and emotional comfort before sleep.

Useful judgment:

> 边界不清，产品就会越做越散，营销也会越讲越虚。

---

## 11. Define Must-have Features

Must-have features are features that directly support the core value anchor.

Ask:

- Without this feature, does the product lose its core value?
- Does this feature serve the primary scenario?
- Does it improve first-use value?
- Does it improve retention?
- Does it improve trust?
- Does it reduce user friction?
- Does it support willingness to pay?
- Is it technically feasible?
- Is the cost justified?

Must-have features should be few.

Typical must-have categories:

- Core function
- Safety mechanism
- Trust-building function
- First-use guidance
- Basic control
- Feedback mechanism
- Maintenance requirement
- Critical app flow
- Essential data display

Useful judgment:

> Must-have 不是团队想做，而是不做就撑不起核心价值。

---

## 12. Define the Must-not-do List

The must-not-do list protects focus.

Ask:

- What features will dilute positioning?
- What features add cost but not value?
- What features increase complexity without improving willingness to pay?
- What features create certification or safety risk?
- What features distract from the core scenario?
- What features are only there because competitors have them?
- What features are good for marketing but bad for experience?
- What features make the product harder to explain?
- What features should wait for the next version?

Common must-not-do examples:

- Add AI without a clear user task.
- Add app control if the core use should be screen-free.
- Add health claims without evidence or certification.
- Add too many modes that confuse first use.
- Add premium materials that increase cost but not perceived value.
- Add features only to look advanced.
- Add multiple scenarios before the primary scenario is validated.

Useful judgment:

> 好产品定义不只写“做什么”，还要明确“什么不做”。

---

## 13. Prioritization Logic

Prioritize by value, evidence, and risk.

Do not prioritize by internal excitement.

A practical priority order:

1. Core value feature
2. Safety and trust feature
3. First-use experience
4. Retention driver
5. Technical risk reduction
6. Cost-sensitive requirement
7. Marketing support feature
8. Nice-to-have feature

Evaluate each feature by:

- User value
- Scenario relevance
- Purchase impact
- Retention impact
- Trust impact
- Technical risk
- BOM impact
- Certification impact
- After-sales impact

Useful judgment:

> 功能优先级不是看谁声音大，而是看谁最支撑核心价值、最影响成败。

---

## 14. Define KXI

KXI means Key Experience Indicator.

It is the key user-perceived signal that proves the product is working.

Ask:

- What should users feel in the first 10 seconds?
- What should users feel in the first use?
- What makes users trust the product?
- What makes users repeat use after 7 days?
- What makes users tell others?
- What makes users think “this is better than my old solution”?

KXI examples:

### Bottle Washer

User sees bottles cleaned, dried, and ready without manual washing.

### Sleep Companion

User feels calmer and less alone during the bedtime transition.

### Blood Flow Wearable

User sees a body signal that helps explain symptoms other wearables do not capture.

### Hair Growth Device

User feels the device is safe, credible, comfortable, and easy to use consistently.

### Smart Helmet

User feels more visible and safer without extra cognitive load.

Useful judgment:

> 没有 KXI，产品就很难从“买了”走到“离不开”。

---

## 15. Translate Product Definition into Hardware Requirements

For hardware products, product definition must become measurable hardware requirements.

Translate:

- Comfort into size, weight, material, temperature, pressure, and wearing time
- Safety into lock mechanisms, sensors, materials, certification, and fail-safe logic
- Cleaning convenience into structure, removable parts, waterproofing, and maintenance time
- Trust into status feedback, transparent process, data display, and certification
- Speed into cycle time, response time, setup time, and recovery time
- Premium perception into CMF, finishing, tolerance, packaging, and interaction quality

Ask:

- Which user value maps to which hardware requirement?
- Which requirement affects purchase?
- Which affects trust?
- Which affects retention?
- Which affects cost?
- Which affects reliability?
- Which affects certification?
- Which affects after-sales?

Useful judgment:

> 用户价值如果不能翻译成硬件指标，最后就会变成研发、供应链和设计各自猜。

---

## 16. MVP Is Minimum Evidence, Not Minimum Product

For hardware, MVP does not always mean a cheap complete product.

MVP means the smallest test that produces useful evidence.

Ask:

- What is the riskiest assumption?
- What is the fastest way to test it?
- Can we test the value without building full hardware?
- Can we test the scenario with a prototype?
- Can we test pricing with a landing page?
- Can we test interaction with a mockup?
- Can we test comfort with a low-fidelity model?
- Can we test demand with ads?
- Can we test trust with concept materials?
- Can we test retention with a diary study?

MVP types:

- Concept test
- Landing page test
- Ad test
- Price test
- Foam model
- Functional prototype
- Wizard-of-Oz prototype
- 3D-printed prototype
- App mockup
- Video demo
- Crowdfunding pre-launch page
- User diary study
- In-home test
- Expert review
- BOM feasibility estimate

Useful judgment:

> MVP 不是做一个缩水版产品，而是用最小成本拿到关键证据。

---

## 17. Validation Plan

A product definition must include a validation plan.

Validate in layers.

### Problem Validation

Ask:

- Does the problem exist?
- Who has it most painfully?
- How often does it happen?
- What do users do today?
- Are users willing to change behavior?

Methods:

- User interviews
- Observation
- Review mining
- Forum / community research
- Search data
- Complaint analysis

### Value Proposition Validation

Ask:

- Do users understand the value?
- Which message converts best?
- What wording resonates?
- What use case is strongest?

Methods:

- Landing page
- Ad testing
- Concept cards
- Message testing
- Survey with forced trade-offs

### Product Experience Validation

Ask:

- Does the prototype deliver perceived value?
- Is first use clear?
- Is the interaction natural?
- Does the product fit the real environment?

Methods:

- Prototype test
- Usability test
- In-home test
- Diary study
- Task completion test

### Price Validation

Ask:

- What price feels acceptable?
- What price feels premium but reasonable?
- What price breaks conversion?
- Does the value anchor justify the price?

Methods:

- Price ladder test
- Landing page A/B test
- Pre-order test
- Van Westendorp
- Conjoint if needed

### Technical Validation

Ask:

- Can the core function be achieved?
- What are the hardest engineering risks?
- Can the product meet requirements?

Methods:

- EVT prototype
- Component testing
- Sensor testing
- Algorithm validation
- Reliability pre-test

### Business Validation

Ask:

- Can CAC support margin?
- Can channel sell it?
- Can reviews support conversion?
- Can the product scale?

Methods:

- Pre-launch campaign
- Crowdfunding test
- CAC test
- Retail buyer feedback
- Channel feedback
- Cost model

Useful judgment:

> 验证不是问用户喜不喜欢，而是用证据打掉最大的不确定性。

---

## 18. Go / No-Go Criteria

A product should not move forward only because the team likes it.

Define evidence thresholds.

Possible Go criteria:

- Users clearly recognize the problem
- Primary scenario is validated
- Value proposition is understood quickly
- Prototype delivers visible KXI
- Price acceptance is within target range
- BOM supports margin
- Key technical risks are controllable
- Certification path is clear
- Supply chain has feasible solution
- Early conversion signal is strong
- User retention signal appears
- Trust cost can be reduced

Possible No-Go criteria:

- Problem is weak or low-frequency
- Users like the idea but do not pay
- Value proposition requires too much explanation
- Core experience is not felt in first use
- Price sensitivity is too high
- BOM cannot support margin
- Technical risk is too high
- Certification blocks claims
- Product requires too much behavior change
- Better alternatives already exist
- Channel cannot sell it
- After-sales risk is unacceptable

Useful judgment:

> Go / No-Go 不是拍脑袋，而是提前定义什么证据能让项目继续，什么证据说明该停。

---

## 19. PRD Definition Logic

A PRD should not be a feature wish list.

A strong PRD should include:

- Product background
- Target user
- Core scenario
- Problem definition
- Existing alternatives
- Value proposition
- Product boundary
- Feature priority
- User flow
- Hardware requirements
- Software requirements
- Interaction requirements
- Safety requirements
- Data requirements
- Performance requirements
- Cost target
- Certification requirements
- Reliability requirements
- Success metrics
- Validation plan
- Risks and assumptions
- Open questions

Useful judgment:

> PRD 的价值不是让研发知道要做什么，而是让所有人知道为什么做、做到什么程度、什么先不做。

---

## 20. Charter Logic

A product charter is not an approval form.

It is a pre-risk-reduction document.

A good charter should clarify:

- What opportunity are we entering?
- Why now?
- Who is the target user?
- What is the core problem?
- What is the proposed solution?
- What is the value anchor?
- What is the business logic?
- What are the key assumptions?
- What risks must be reduced before full development?
- What evidence is required for the next gate?
- What resources are needed?
- What is the Go / No-Go condition?

Useful judgment:

> Charter 不是为了立项而写，而是为了在立项前把最危险的假设暴露出来。

---

## 21. Product Strategy Five-step Lens

When defining product strategy, use this five-step lens:

### 1. Demand

What user problem are we solving?

### 2. Solution

What product solution creates clear value?

### 3. Business Model

How does the product make money sustainably?

### 4. Growth Mechanism

How does the product acquire users and spread?

### 5. Defensibility

Why will competitors not easily take the opportunity away?

Ask:

- Is demand strong enough?
- Is the solution differentiated?
- Is the business model healthy?
- Is growth repeatable?
- Is defensibility real?

Useful judgment:

> 产品策略不是“我要做什么功能”，而是需求、解法、商业、增长、壁垒五件事能不能互相咬住。

---

## 22. Macro-Meso-Micro-Single Point Method

Use this method when moving from research to product definition.

### Macro

Understand broad direction:

- Social trend
- Technology trend
- Regulation
- Demographics
- Economic environment
- Category shift

Macro gives direction.

### Meso

Find category opportunity:

- Category stage
- Competitive landscape
- Price bands
- Channel structure
- User education level
- Brand gaps

Meso finds opportunity space.

### Micro

Understand user and scenario:

- User task
- Pain point
- Current alternative
- Trigger moment
- Usage environment
- Decision logic

Micro defines product direction.

### Single Point

Find the breakthrough point:

- Hard value anchor
- Killer feature
- KXI
- Non-consensus insight
- Differentiated entry point

Single point creates breakthrough.

Useful judgment:

> 宏观看方向，中观看机会，微观看用户，单点打突破。

---

## 23. User Journey for Product Definition

When defining a product from user journey, map:

- Before the need appears
- Trigger moment
- Current behavior
- Friction point
- Product intervention
- First-use experience
- Success moment
- Repeat-use reason
- Share or referral moment
- Failure or drop-off moment

For each step, analyze:

- User action
- User thought
- User emotion
- Environment
- Current tool
- Pain point
- Opportunity
- Product requirement
- Evidence needed

Useful judgment:

> 用户旅程不是画流程图，而是找到产品能在哪个瞬间真正改变用户行为。

## 24. Example: Bedtime Companion Product Definition Lens

When defining bedtime-companion-like products, use this type of logic.

### Target User

Users who repeatedly experience bedtime anxiety, loneliness, racing thoughts, body tension, or phone-scrolling before sleep.

### Core Scenario

The user enters bed but cannot transition into rest. They need calming, emotional support, and body-based comfort without more screen stimulation.

### Current Alternatives

- Phone scrolling
- Music
- Meditation app
- Weighted blanket
- Plush toy
- Partner companionship
- Sleep stories
- Breathing exercises
- Doing nothing

### Unmet Need

A screen-free, embodied, emotionally warm transition ritual before sleep.

### Value Anchor

Physical comfort plus gentle companionship that helps the user shift from high arousal to rest.

### Must-have

- Soft body
- Hug interaction
- Breathing rhythm
- Warmth or pressure cue
- Gentle conversation
- Low-stimulation bedtime mode
- Simple control
- Safe materials

### Must-not-do

- Overstimulating conversation
- Too much screen interaction
- Medical insomnia claims
- Complex app dependency
- Too many modes before core use is validated

### KXI

User feels calmer, less alone, and more willing to put the phone down during the bedtime transition.

Useful judgment:

> 睡前陪伴硬件的核心不是 AI 聊天，而是用具身互动帮助用户完成睡前状态切换。

---

## 25. Example: Bottle Washer Product Definition Lens

When defining bottle washer-like products, use this type of logic.

### Target User

Parents or caregivers who repeatedly clean baby bottles and feeding accessories, especially under fatigue and time pressure.

### Core Scenario

After feeding, the caregiver must clean bottles quickly and reliably while dealing with baby care, sleep deprivation, and household tasks.

### Current Alternatives

- Hand washing
- Dishwasher
- Sterilizer
- Dryer
- Separate cleaning tools
- Asking family members to help

### Unmet Need

Reduce repetitive cleaning burden and trust that bottles are clean, dry, and ready.

### Value Anchor

Less manual work plus more confidence in bottle hygiene.

### Must-have

- Effective washing
- Drying
- Sterilization if relevant
- Easy loading
- Easy cleaning
- Low maintenance
- Clear status feedback
- Reasonable cycle time
- Bottle compatibility

### Must-not-do

- Add smart features that do not improve cleaning confidence
- Overcomplicate operation
- Create high maintenance burden
- Increase price without hard value improvement
- Make the device too large for kitchen counters

### KXI

User sees bottles come out clean, dry, and ready with minimal manual intervention.

Useful judgment:

> 奶瓶清洗产品的核心不是“智能”，而是多快好省地把瓶子洗干净，并让用户放心。

---

## 26. Product Definition Output Template

When defining a new product, use this structure when appropriate:

1. One-sentence product definition
2. Target user
3. Core scenario
4. Current alternative
5. Unmet need
6. Value proposition
7. Value anchor
8. Purchase trigger
9. Product boundary
10. Must-have features
11. Must-not-do list
12. KXI
13. Hardware requirements
14. MVP / prototype
15. Pricing hypothesis
16. Validation plan
17. Go / No-Go criteria
18. Final recommendation

Do not force every section if the user asks for a short answer.

---

## 27. Final Product Definition Rule

When uncertain, use this rule:

A product definition is strong only when it can clearly answer:

- Who exactly is this for?
- What specific scenario does it serve?
- What painful or valuable task does it solve?
- Why are current alternatives not enough?
- What is the hard value anchor?
- What is the purchase trigger?
- What is the first-use KXI?
- What is the 7-day retention reason?
- What must be built first?
- What must not be built yet?
- What evidence proves this is worth continuing?

If these cannot be answered, the product is not yet defined.

It is still only an idea.
